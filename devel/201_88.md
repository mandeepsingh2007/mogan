# 201_86 修复替换全部在注入型替换下卡死与提前结束

## 如何测试
1. 打开`TeXmacs/tests/tmu/201_86.tmu`
2. 选中一个a，打开替换窗口，替换为 `axa`，执行“替换全部”。
3. 预期：
   - 不会卡死/无响应；
   - 每个原始 `a` 仅替换一次（如 `aaa -> axaaxaaxa`，`aa -> axaaxa`）。

## 2026/03/03 修复 replace-all 在注入型替换场景下的游标推进
### What
- 修改 `TeXmacs/progs/generic/search-widgets.scm` 中 `replace-next`：
  - 新增 `replace-next-selection-after-start` 与 `replace-move-to-next-after`，按 `start >= mid-p` 选择下一命中；
  - 新增 `replace-search-next-after`，在单次替换后以 `mid-p` 为参考点执行同步搜索推进；
  - 当当前搜索上限找不到 `mid-p` 之后的命中时，按 `100 -> 200 -> 400 -> ...` 逐步扩大 `limit`；
  - 直到找到下一命中，或达到 `replace-search-max-limit`（`1000000`）后停止。
  - `replace-next` 仅在“成功推进到下一命中且光标前进”时返回 `#t`。

### Why
- 旧逻辑在 `a -> axa` 这类“替换文本包含原搜索词”的场景中，会重新命中新插入文本，可能导致 replace-all 长时间循环甚至卡死。
- 固定 `limit=100` 的搜索窗口在高密度命中场景下会提前耗尽结果，导致后续段落未被替换（replace-all 提前结束）。

### How
- 通过 `set-search-reference mid-p` 固定下一轮搜索起点，避免回卷到已替换区间。
- 在 `replace-search-next-after` 中同步调用 `perform-search-sub`，并通过 `replace-move-to-next-after` 选择首个 `start >= mid-p` 的命中。
- 若没找到下一命中则继续扩容 `limit`，直到命中或达到上限。
- 避免依赖异步扩容，确保 replace-all 在大文本中也能持续向后推进，不因结果截断提前停止。
